# Commit Changes

このスキルは、Git リポジトリにおける変更内容の把握から、コミット、プッシュまでの一連のワークフローを支援します。

## 運用ルール

### 1. コミットの粒度
- 変更は論理的な単位（1つの機能追加や1つのバグ修正など）で分割してコミットすることを推奨します。
- 大規模なリファクタリングと新機能の追加は、可能な限り別のコミットに分けてください。

### 2. コミットメッセージの作成 (Conventional Commits)
変更内容に基づき、以下の **Conventional Commits** 形式でメッセージを作成します。

**形式:** `<type>(<scope>): <description>`

- **type**:
  - `feat`: 新機能の追加
  - `fix`: バグ修正
  - `docs`: ドキュメントのみの変更
  - `style`: コードの意味に影響を与えない変更（ホワイトスペース、フォーマット、セミコロンの欠落など）
  - `refactor`: バグ修正や機能追加を行わないコードの変更
  - `perf`: パフォーマンスを向上させるための変更
  - `test`: 不足しているテストの追加や既存のテストの修正
  - `chore`: ビルドプロセスやドキュメント生成などの補助ツール、ライブラリの変更
- **scope (省略可能)**: 変更が及ぶ範囲（例: `analysis`, `pipelines`, `data`, `skill` など）。
- **description**: 変更内容の簡潔な要約（日本語を推奨）。「何をしたか」だけでなく、必要に応じて「なぜしたか」が伝わるように記述します。

**例:**
- `feat(analysis): 骨格解析に新しいモデルを追加`
- `fix(pipelines): 動画ダウンロードのタイムアウトを修正`
- `docs: README.md にセットアップ手順を追記`
- `chore(skill): commit-changes スキルのメッセージルールを更新`

### 3. セキュリティと安全
- `git add` 前に必ず変更内容を確認し、APIキー、パスワード、秘密鍵などの機密情報が含まれていないことを確認してください。
- 誤って機密情報をステージングした場合は、`git reset` を使用して取り消してください。

### 4. プッシュと同期
- リモートへのプッシュ前に `git pull` を行い、競合が発生しないか確認することを推奨します。
- 原則として `main` ブランチへの直接プッシュは避け、開発用ブランチでの作業を検討してください（プロジェクトの運用方針に従います）。

## ワークフロー

### 1. 変更内容の分析
まず、現在のリポジトリの状態を確認します。
- `git status` を実行して、追跡対象外のファイルやステージングされていない変更を確認します。
- `git diff`（または `git diff --cached`）を実行して、具体的なコードの変更内容を確認します。

### 2. ステージングとコミット
- `git add <files>` で変更をステージングします。
- `git commit -m "<message>"` でコミットを作成します。

### 3. プッシュ
- `git push origin <branch_name>` でリモートリポジトリに変更を送信します。

## 使用例

### 例1: 変更内容を分析してコミットする
ユーザー: 「今の変更をコミットしてプッシュして」
Gemini: `git status` と `git diff` で内容を確認し、`feat(analysis): ...` のようなメッセージを提案して実行します。

## 注意事項
- 秘密情報（APIキーなど）が含まれていないか、常に確認してください。
- コミットメッセージは、後で履歴を振り返った時に「何のために行った変更か」が明確にわかるように記述してください。
